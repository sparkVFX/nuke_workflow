# 2026-03-26

## 修复 nanobanana.py 和 veo.py 提示词持久化问题

**问题**：`NanoBananaWidget` 和 `VeoWidget` 中的 `_save_prompts_to_node()` / `_restore_prompts_from_node()` 使用全局搜索函数 (`get_nanobanana_node()` / `_get_veo_node()`) 查找节点，依赖 `nuke.selectedNodes()`。当用户切换到其他节点再切回来时，Nuke 会销毁旧 widget 并重建新 widget，此时全局搜索可能找不到正确的节点，导致提示词恢复失败。

**修复方案（第一轮）**：
- `NanoBananaWidget` 和 `VeoWidget` 增加 `node` 参数，构造时缓存为 `self._node`
- 新增 `_get_owner_node()` 辅助方法，优先使用缓存的 `self._node`，失效时 fallback 到 `nuke.thisNode()`
- `NanoBananaKnobWidget` 和 `VeoKnobWidget` 在构造时通过 `nuke.thisNode()` 获取当前节点并传递
- 所有内部方法改用 `self._get_owner_node()`
- 删除了 `veo.py` 中旧的 `_get_veo_node()` 方法

## 第二轮修复：保存/恢复所有 UI 参数（不止提示词）

**问题**：第一轮只保存了 prompt 和 neg_prompt，但模型选择、比例、分辨率、种子、模式等参数在切换节点后仍会丢失。

**修复方案**：
- 用 `_save_all_state_to_node()` / `_restore_all_state_from_node()` 替换旧的只保存提示词的方法
- 所有 UI 状态序列化为 JSON 存入单个隐藏 knob（`nb_ui_state` / `veo_ui_state`）
- **nanobanana.py** 保存：model_index, ratio_index, resolution_index, seed_random, seed_value, prompt, neg_prompt
- **veo.py** 保存：model_index, ratio_index, resolution_index, duration_index, mode_index, prompt_mode_index, prompt, neg_prompt
- 为所有控件（ComboBox、CheckBox、LineEdit、TextEdit）都连接了 `_save_all_state_to_node` 信号
- 添加 `hideEvent` 和 `closeEvent` 保证 widget 销毁/隐藏前必定保存
- 恢复时 `blockSignals(True)` 避免触发循环保存

## 第三轮修复：参数依然无法保留

**问题根因分析**：
1. `String_Knob` 有 ~256 字符长度限制，JSON 序列化的状态字符串很容易超出被截断
2. Nuke 的 `PyCustom_Knob` 在切换节点时**直接销毁 C++ 层 widget**，不一定触发 Qt 的 `hideEvent`/`closeEvent`
3. `_get_owner_node()` 中的 `nuke.thisNode()` fallback 在信号回调时可能返回错误节点

**修复方案**：
- **不再使用 JSON 单 knob**，改为每个参数使用独立的隐藏 knob：
  - nanobanana: `nb_s_model`(Int), `nb_s_ratio`(Int), `nb_s_res`(Int), `nb_s_seed_rnd`(Boolean), `nb_s_seed_val`(Text), `nb_s_prompt`(Text), `nb_s_neg`(Text)
  - veo: `veo_s_model`(Int), `veo_s_ratio`(Int), `veo_s_res`(Int), `veo_s_dur`(Int), `veo_s_mode`(Int), `veo_s_pm`(Int), `veo_s_prompt`(Text), `veo_s_neg`(Text)
- 提示词使用 `Multiline_Eval_String_Knob`（无长度限制）
- `_save_all_state_to_node()` 中直接使用 `self._node` 而非 `_get_owner_node()` 的 fallback
- **`updateValue()` 方法**：Nuke 在 panel 可见时每次 knob 变化都会调用此方法，利用它来保存状态（这是最可靠的保存时机）
- 添加 `event()` 覆写捕获 `QEvent.DeferredDelete`（widget 被延迟删除时的最后保存机会）
- 添加 `[NanoBanana]`/`[VEO]` 前缀的调试日志

## VEO 输入节点根据模式重命名

**需求**：根据不同模式，VEO_Generate Group 内部的 Input 节点需要有有意义的名称（而非默认的 A1/A2/A3）。

**实现**：修改 `_update_node_inputs(mode)` 方法：
- FirstFrame 模式（1个输入）：Input 命名为 `FirstFrame`

- Ingredients 模式（3个输入）：命名为 `img1`、`img2`、`img3`

- 按 xpos 排序保证稳定的索引顺序

## 删除 DEBUG 代码块，恢复 API 调用

**问题**：nanobanana.py (第1943-1947行) 和 veo.py (第283-288行) 中存在激活状态的 DEBUG 代码块，在 API 调用前 `return`，导致所有生成请求都被跳过不会真正发送到 Google。

**关于输出路径**：NanoBanana 的输出路径已经通过 `get_output_directory()` → `get_temp_directory()` → `NanoBananaSettings.temp_directory` 正确关联用户设置的 temp 路径。如果用户没有在 Setting 中自定义，则默认使用 `%TEMP%/nanobanana_temp/output`。

## 修复自定义 temp 路径不生效的问题

**问题**：用户在 Setting 中设置了 `E:/BaiduNetdiskDownload/nuke_workflow/temp` 作为临时目录，但 Effective path 仍然显示为 `C:\Users\...\AppData\Local\Temp\nanobanana_temp`，实际的 input/output 文件夹也在 C 盘默认位置。

**根因**：`NanoBananaSettings.temp_directory` getter (第272行) 和 `_update_effective_path` (第406行) 都用 `os.path.isdir(custom_dir)` 检查目录是否存在——如果用户指定的路径尚未创建，就会回退到系统默认路径。

**修复**：
1. `temp_directory` getter：当用户指定了路径但目录不存在时，自动 `os.makedirs()` 创建，而不是回退
2. `_update_effective_path`：只要用户填写了路径就显示该路径作为 effective path，不再要求目录已存在

## 修复 NanoBanana API 未传递 aspect_ratio 和 resolution

**问题**：用户选择的比例（如 16:9）和分辨率（如 2K）虽然传入了 `NanoBananaWorker`，但在构建 `generation_config` 时只包含了 `responseModalities`，**完全没有传给 Gemini API**。这两个值仅用于日志记录。

**修复**：在 `NanoBananaWorker.run()` 的 `generation_config` 中增加 `imageConfig` 对象：
```python
generation_config["imageConfig"] = {
    "aspectRatio": "16:9",   # 来自 self.aspect_ratio（"Auto" 时不传）
    "imageSize": "2K"        # 来自 self.resolution
}
```
符合 Gemini REST API 文档要求的格式。

## 修复 Widget 销毁后 Worker 完成回调丢失的 bug

**问题**：当用户在 NanoBanana/VEO 面板中点击生成后，切换到其他节点（面板被销毁），后台 Worker 完成后无法创建 Prompt/Read 节点。原因：
1. Worker 作为 `self.current_worker` 存储在 Widget 上，Widget 销毁后 Worker 可能被 GC
2. `_on_gen_finished` 回调中通过 `self._gen_params` 访问参数，但 `self` 已被销毁
3. Qt 信号连接到已销毁 Widget 的方法上，信号发射后可能无法触发

**修复**（同时应用到 nanobanana.py 和 veo.py 的所有 4 个 Widget 类）：
1. 新增模块级字典 `_active_workers` / `_veo_active_workers`，用于持有正在运行的 Worker（防止 GC）
2. 将回调从实例方法改为闭包函数，通过闭包捕获 `gen_params`、`worker_id`、`widget_ref` 等引用
3. 闭包中使用 `_isValid(widget_ref)` 安全检查 Widget 是否存活，存活则更新 UI，否则静默跳过
4. 无论 Widget 是否存活，都会执行核心逻辑（创建节点 / 更新 Read 节点）
5. Worker 完成或出错后从模块级字典中清理引用

**影响的类**：
- `NanoBananaWidget._start_generate` → 闭包 `_on_finished` / `_on_error`
- `NanoBananaPromptWidget._start_generate` → 闭包 `_on_finished` / `_on_error`
- `VeoWidget._start_generate` → 闭包 `_on_finished` / `_on_error`
- `VeoRecordWidget._regenerate` → 闭包 `_on_finished` / `_on_error`
